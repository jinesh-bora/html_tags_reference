Semantic Tags

HTML elements that convey meaning about the structure and content of a web page

  • Back to Main Reference

About Semantic HTML

Semantic HTML uses elements that clearly describe their meaning to both the browser and the developer. These elements provide information about the content they contain, creating a more structured, accessible, and SEO-friendly document.

Prior to HTML5, developers relied heavily on <div> and <span> elements with class or ID attributes to structure web pages. HTML5 introduced a set of semantic elements that better describe different parts of a web page, making code more meaningful and easier to understand.

Benefits of Semantic HTML

  • Accessibility: Screen readers and assistive technologies can better interpret the page content
  • SEO: Search engines can better understand the structure and importance of content
  • Readability: Code becomes more readable and maintainable
  • Consistency: Promotes a standardized approach to web page structure
  • Mobile-friendliness: Helps with responsive design and content prioritization
  • Future-proof: As browsers and technologies evolve, semantic markup helps ensure compatibility

Page Structure Elements

<header>

Represents a container for introductory content or navigational aids. Typically contains headings, logos, navigation, and search forms.

Example:

<header>
  <h1>Website Name</h1>
  <nav>
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/about">About</a></li>
      <li><a href="/contact">Contact</a></li>
    </ul>
  </nav>
</header>

The <header> element is typically used at the beginning of the document, but can also be used within other semantic elements like <article> or <section> to define a header for that content.

<nav>

Defines a section containing navigation links. Should be used for major navigation blocks, not for all groups of links.

Example:

<nav>
  <ul>
    <li><a href="/">Home</a></li>
    <li><a href="/products">Products</a></li>
    <li><a href="/services">Services</a></li>
    <li><a href="/about">About Us</a></li>
    <li><a href="/contact">Contact</a></li>
  </ul>
</nav>

Using <nav> helps screen readers identify navigation sections and allows users to skip directly to or past the navigation.

<main>

Specifies the main content of a document. There should be only one <main> element per page, and it should not be nested inside <article>, <aside>, <footer>, <header>, or <nav>.

Example:

<main>
  <h1>Main Article Title</h1>
  <p>Introduction to the main content...</p>
  
  <section>
    <h2>First Section</h2>
    <p>Content for the first section...</p>
  </section>
  
  <section>
    <h2>Second Section</h2>
    <p>Content for the second section...</p>
  </section>
</main>

The <main> element helps screen readers and other assistive technologies identify the primary content of the page.

<article>

Represents a self-contained composition in a document, page, application, or site that is intended to be independently distributable or reusable.

Examples include:

  • Blog posts
  • News articles
  • Forum posts
  • User comments
  • Product cards
  • Any independent content

Example:

<article>
  <header>
    <h2>Article Title</h2>
    <p>Posted by <a href="#">Author</a> on <time datetime="2025-06-04">June 4, 2025</time></p>
  </header>
  
  <p>Introduction to the article...</p>
  
  <section>
    <h3>Article Section</h3>
    <p>Content for this section...</p>
  </section>
  
  <footer>
    <p>Tags: <a href="#">HTML5</a>, <a href="#">Semantic</a></p>
  </footer>
</article>

An <article> should make sense on its own, even if all surrounding content is removed.

<section>

Represents a standalone section of content that is thematically related. Typically contains a heading.

Example:

<section>
  <h2>Features</h2>
  <p>Introduction to our product features...</p>
  
  <div class="feature-grid">
    <div class="feature">
      <h3>Feature 1</h3>
      <p>Description of feature 1...</p>
    </div>
    
    <div class="feature">
      <h3>Feature 2</h3>
      <p>Description of feature 2...</p>
    </div>
  </div>
</section>

The difference between <section> and <article> is that <section> groups related content, while <article> is for self-contained, independently distributable content.

<aside>

Represents content that is tangentially related to the content around it, which could be considered separate from that content. Often displayed as sidebars or call-out boxes.

Example:

<main>
  <article>
    <h1>Main Article Title</h1>
    <p>Main article content...</p>
  </article>
  
  <aside>
    <h2>Related Articles</h2>
    <ul>
      <li><a href="#">Related Article 1</a></li>
      <li><a href="#">Related Article 2</a></li>
    </ul>
    
    <h2>Advertisement</h2>
    <div class="ad-container">
      <!-- Ad content -->
    </div>
  </aside>
</main>

Common uses for <aside> include sidebars, pull quotes, advertising, groups of navigation elements, and other content that is separate from the main content.

<footer>

Represents a footer for its nearest sectioning content or sectioning root element. Typically contains information about the author, copyright data, links to related documents, etc.

Example:

<footer>
  <div class="footer-links">
    <div>
      <h3>Company</h3>
      <ul>
        <li><a href="/about">About Us</a></li>
        <li><a href="/careers">Careers</a></li>
      </ul>
    </div>
    
    <div>
      <h3>Support</h3>
      <ul>
        <li><a href="/contact">Contact Us</a></li>
        <li><a href="/faq">FAQ</a></li>
      </ul>
    </div>
  </div>
  
  <div class="copyright">
    <p>© 2025 Company Name. All rights reserved.</p>
  </div>
</footer>

Like <header>, the <footer> element can be used within other semantic elements to provide footer information for that specific content.

Additional Semantic Elements

<figure> and <figcaption>

Used to encapsulate media content with an optional caption.

Example:

<figure>
  <img src="diagram.jpg" alt="Project workflow diagram">
  <figcaption>Fig. 1: Workflow diagram showing the development process.</figcaption>
</figure>

The <figure> element is not limited to images. It can be used with videos, code blocks, illustrations, diagrams, and other self-contained content.

<time>

Represents a specific period in time. It can include the datetime attribute to translate dates into a machine-readable format.

Example:

<p>The event starts at <time datetime="2025-07-15T19:00:00">7:00 PM on July 15, 2025</time>.</p>

<p>Posted on <time datetime="2025-06-04">June 4, 2025</time></p>

The datetime attribute follows a specific format (ISO 8601) that helps machines parse the date and time information.

<mark>

Represents text that is highlighted or marked for reference or notation purposes.

Example:

<p>The most important part of the instructions is <mark>never press the red button</mark>.</p>

<p>Search results for "semantic HTML": We found several instances of <mark>semantic HTML</mark> in the documentation.</p>

The <mark> element is useful for highlighting search terms, emphasizing content temporarily relevant to the user, or highlighting code examples.

<address>

Provides contact information for its nearest <article> or <body> ancestor.

Example:

<address>
  Written by <a href="mailto:john@example.com">John Doe</a>.<br>
  Visit us at:<br>
  Example.com<br>
  123 Main St.<br>
  Anytown, CA 12345<br>
  USA
</address>

The <address> element is not just for physical addresses—it can include any type of contact information, such as email, phone number, social media handles, or physical address.

<details> and <summary>

Creates an interactive widget that the user can open and close. The <summary> element provides a visible heading for the <details> element.

Example:

<details>
  <summary>Click to show more information</summary>
  <p>This is additional information that is hidden by default. It becomes visible when the user clicks on the summary.</p>
  <p>You can include any HTML content inside the details element.</p>
</details>

The <details> element is useful for FAQs, accordion-style content, or any information that should be revealed on demand.

Comparison with Non-Semantic Elements

Non-Semantic Approach

<div class="header">
  <h1>Website Title</h1>
  <div class="nav">
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/about">About</a></li>
    </ul>
  </div>
</div>

<div class="main-content">
  <div class="article">
    <h2>Article Title</h2>
    <div class="article-content">
      <p>Article content...</p>
    </div>
  </div>
  
  <div class="sidebar">
    <h3>Related Links</h3>
    <ul>
      <li><a href="#">Link 1</a></li>
      <li><a href="#">Link 2</a></li>
    </ul>
  </div>
</div>

<div class="footer">
  <p>Copyright 2025</p>
</div>

Semantic Approach

<header>
  <h1>Website Title</h1>
  <nav>
    <ul>
      <li><a href="/">Home</a></li>
      <li><a href="/about">About</a></li>
    </ul>
  </nav>
</header>

<main>
  <article>
    <h2>Article Title</h2>
    <div class="article-content">
      <p>Article content...</p>
    </div>
  </article>
  
  <aside>
    <h3>Related Links</h3>
    <ul>
      <li><a href="#">Link 1</a></li>
      <li><a href="#">Link 2</a></li>
    </ul>
  </aside>
</main>

<footer>
  <p>Copyright 2025</p>
</footer>

The semantic approach:

  • Conveys the meaning and purpose of each section
  • Makes the HTML more readable and easier to understand
  • Provides better accessibility for assistive technologies
  • Helps search engines better understand the content structure
  • Reduces the need for descriptive class names

Accessibility Benefits

Semantic HTML significantly improves accessibility in several ways:

Screen Reader Navigation

Screen readers can use semantic elements to:

  • Identify key parts of a page (header, main content, navigation, etc.)
  • Allow users to jump directly to specific sections
  • Provide better context about the content structure
  • Differentiate between primary and supplementary content

Keyboard Navigation

Proper semantic structure improves keyboard navigation by:

  • Creating logical tab order
  • Providing clear focus indicators
  • Enabling navigation shortcuts in some screen readers

Assistive Technologies

Beyond screen readers, semantic HTML helps:

  • Voice recognition software identify interactive elements
  • Translation tools understand content structure
  • Browser reading modes to display content appropriately
  • Search engines extract and display relevant content

Best Practices for Semantic HTML

  • Use the right element for the right purpose: Choose elements based on their meaning, not their default appearance
  • Don't overuse <div> and <span>: Use semantic elements whenever possible
  • Create a clear document outline: Use headings (<h1> through <h6>) in a logical hierarchy
  • Use <main> for primary content: Include only one <main> element per page
  • Distinguish between <article> and <section>: Use <article> for self-contained content and <section> for thematically related content
  • Add alternative text: Always include alt attributes for images
  • Use <button> for interactive elements: Don't use <div> or other elements with JavaScript to create buttons
  • Use <table> only for tabular data: Never use tables for layout purposes
  • Include appropriate ARIA attributes: Enhance accessibility where native semantics aren't sufficient
  • Test with screen readers: Verify that your semantic structure works well with assistive technologies

When to Use Semantic Elements vs. div

Use semantic elements when:

  • The content has a specific meaning or purpose that matches a semantic element
  • The content represents a distinct section of the page
  • The content would benefit from being identified by assistive technologies

Use <div> when:

  • The element is purely for styling or layout purposes
  • No existing semantic element accurately represents the content's purpose
  • You need a generic container for JavaScript functionality
  • You're creating a component that doesn't fit any semantic element's description
